# v2.5 律师工作台实施计划

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` (recommended) or `superpowers:executing-plans` to implement this plan task-by-task. Steps use checkbox (`- [x]`) syntax for tracking.

**Goal:** 在现有对抗分析引擎之上，交付律师工作台层——案件列表/回放/复核/导出，以及用户级权限与审计。

**Architecture:** 每个 Phase 独立可合并到 main，在 `api/` 层新增模块（`case_index.py`、`event_log.py`、`review_service.py`、`users.py`、`permissions.py`、`export_service.py`）而非重写 `service.py`。状态转换审计通过 append-only JSONL 事件日志记录，持久化到既有 `WorkspaceManager` 管理的工作区目录下。

**Tech Stack:** FastAPI · Pydantic v2 · python-jose[cryptography]（JWT，Phase 4 引入） · pytest · 既有 `WorkspaceManager` / `CaseStore`

---

## v2.5 计划概览

v2.5 是律所部署的前置条件版本。核心诉求：

1. **可见性**：律师必须能列出所有案件，按状态/类型/日期筛选，不依赖事先知道 `case_id`。
2. **可追溯性**：每个状态迁移、引擎调用、人工操作都留下不可篡改的事件轨迹。
3. **可复核性**：分析完成后，律师可以独立标注/批准/打回报告，工作流不绕过人工。
4. **可管控性**：每个操作绑定到具体用户，不同角色的权限边界明确，可审计 who-did-what。
5. **可交付性**：案件数据可以结构化导出（JSON/DOCX/ZIP），满足归档和对外交付需求。

每个 Phase 均可单独合并、单独上线，不依赖后续 Phase 的代码。

```python
# api/schemas.py 新增

class CaseListEntry(BaseModel):
    case_id: str
    status: CaseStatus
    case_type: str
    plaintiff_name: str
    defendant_name: str
    created_at: datetime         # 从 case_meta.json 读取
    updated_at: datetime
    has_report: bool

class CaseListResponse(BaseModel):
    items: list[CaseListEntry]
    total: int
    page: int
    page_size: int

class CaseListQuery(BaseModel):
    status: Optional[CaseStatus] = None
    case_type: Optional[str] = None
    from_date: Optional[datetime] = None
    to_date: Optional[datetime] = None
    page: int = Field(default=1, ge=1)
    page_size: int = Field(default=20, ge=1, le=100)   # 超过 100 → 422
    sort: str = "-created_at"    # 前缀 - 降序；支持字段：created_at, updated_at, status
```

> **注意（FastAPI 查询参数绑定）**：`CaseListQuery` 是 Pydantic `BaseModel`，用于 GET 路由时必须通过 `Depends()` 绑定，否则 FastAPI 会尝试将其解析为请求体。路由签名为：
>
> ```python
> @app.get("/api/cases", response_model=CaseListResponse)
> async def list_cases(q: CaseListQuery = Depends()) -> CaseListResponse:
>     ...
> ```
>
> `Depends()` 不带参数时 FastAPI 自动将 `BaseModel` 字段映射为查询参数。`page_size` 的 `le=100` 约束由 Pydantic 校验，超出时 FastAPI 自动返回 422。

#### 1.5 启动扫描

```python
# api/app.py

from contextlib import asynccontextmanager
from .service import _WORKSPACE_BASE   # _WORKSPACE_BASE 定义在 service.py，需显式导入

@asynccontextmanager
async def lifespan(app: FastAPI):
    # 启动：扫描磁盘重建索引
    if _WORKSPACE_BASE is not None:
        n = case_index.scan_from_disk(_WORKSPACE_BASE)
        logger.info("案件索引重建完成，共 %d 条", n)
    yield
    # 关闭：无清理需要

app = FastAPI(..., lifespan=lifespan)
```

```python
# engines/shared/event_log.py

from enum import Enum
from datetime import datetime, timezone
from pydantic import BaseModel, Field
import uuid

class EventType(str, Enum):
    case_created       = "case_created"
    material_added     = "material_added"
    extraction_started = "extraction_started"
    extraction_done    = "extraction_done"
    extraction_failed  = "extraction_failed"
    confirmed          = "confirmed"
    analysis_started   = "analysis_started"
    analysis_done      = "analysis_done"
    analysis_failed    = "analysis_failed"
    review_submitted   = "review_submitted"   # Phase 3 emit
    exported           = "exported"           # Phase 5 emit

class CaseEvent(BaseModel):
    event_id:   str = Field(default_factory=lambda: f"evt-{uuid.uuid4().hex[:12]}")
    case_id:    str
    event_type: EventType
    actor_id:   str = "system"               # Phase 4 填真实 user_id
    payload:    dict = Field(default_factory=dict)
    created_at: datetime = Field(default_factory=lambda: datetime.now(timezone.utc))

class EventLog:
    """线程安全的 append-only JSONL 事件日志。
    存储路径：{workspace_dir}/{case_id}/events.jsonl
    每行一个 JSON 对象（CaseEvent.model_dump(mode='json')）。

    线程安全策略：使用 per-instance `threading.Lock` 保护追加操作。
    注意：`open(path, 'a')` 在 Windows 上不提供进程间或线程间原子性，
    必须依赖显式 Lock。同一进程内多实例共享同一文件时，每次 append 前
    持锁（实例级锁），写完即释放。生产部署为单进程模式，此策略足够。
    """

    def __init__(self, workspace_dir: Path, case_id: str) -> None:
        self._path = workspace_dir / case_id / "events.jsonl"
        self._lock = threading.Lock()

    def append(self, event: CaseEvent) -> None:
        """线程安全追加。绝不覆盖已有行。"""
        ...

    def load_all(self) -> list[CaseEvent]:
        """返回按 created_at 升序排列的全部事件。"""
        ...

    def load_since(self, after_event_id: str) -> list[CaseEvent]:
        """返回指定 event_id 之后的事件（用于增量轮询）。"""
        ...
```

```python
def append_event(self, event: "CaseEvent") -> None:
    """Append one event to events.jsonl (thread-safe)."""
    from engines.shared.event_log import EventLog
    EventLog(self.base_dir, self.case_id).append(event)

def load_events(self, after_event_id: str | None = None) -> list["CaseEvent"]:
    from engines.shared.event_log import EventLog
    log = EventLog(self.base_dir, self.case_id)
    if after_event_id:
        return log.load_since(after_event_id)
    return log.load_all()
```

- [x] **Step 5b：产物版本快照 — `save_artifact_version()`**

  在 `WorkspaceManager` 中新增方法，覆盖产物前先将旧版本重命名为 `{name}.v{N}.json`（N 自增），再写入新版本：

  ```python
  # engines/shared/workspace_manager.py
  def save_artifact_version(self, name: str, data: dict) -> None:
      """覆盖前重命名旧版本为 {name}.v{N}.json（N 自增），再写新版本。
      适用于 evidence_index.json / issue_tree.json 等可能被多次生成的产物。"""
      artifacts_dir = self.base_dir / self.case_id / "artifacts"
      artifacts_dir.mkdir(parents=True, exist_ok=True)
      target = artifacts_dir / f"{name}.json"
      if target.exists():
          n = 1
          while (artifacts_dir / f"{name}.v{n}.json").exists():
              n += 1
          target.rename(artifacts_dir / f"{name}.v{n}.json")
      target.write_text(json.dumps(data, ensure_ascii=False, indent=2), encoding="utf-8")
  ```

  ```
    evidence_index.v1.json    # 第一次分析产物（覆盖前保留）
    issue_tree.json
    issue_tree.v1.json
    ...
  ```

#### 3.4 端点设计

```

POST /api/cases/{case_id}/reviews
Body: { "action": "pending_review" }
→ 202，案件进入待复核状态

→ 200, ReviewResponse

# 获取复核历史
GET /api/cases/{case_id}/reviews
→ 200, ReviewListResponse
```

**状态机约束**：

| 当前 `review_status` | 可触发的 `action` | 调用者角色（Phase 3 前为"任意"，Phase 4 后强制） |
|---------------------|------------------|------|
| `none` / `revision_requested` | `pending_review` | 律师（`review_submit`） |
| `pending_review` | `approved` / `rejected` / `revision_requested` | 复核员（`review_decide`） |
| `approved` / `rejected` | — | 终态，返回 400 |

**单端点双角色权限设计**：Phase 3 时没有用户角色，服务端按 `action` 值分流校验：

```python
# api/app.py （Phase 3 实现）
@app.post("/api/cases/{case_id}/reviews", response_model=ReviewResponse)
async def submit_review(case_id: str, body: ReviewRequest):
    record = _get_case_or_404(case_id)
    # 校验案件必须已分析
    if record.status != CaseStatus.analyzed:
        raise HTTPException(400, "只有分析完成的案件才能提交复核")
    # 状态机校验
    if body.action == ReviewStatus.pending_review:
        if record.review_status not in (ReviewStatus.none, ReviewStatus.revision_requested):
            raise HTTPException(400, f"当前复核状态 {record.review_status} 不允许提交待复核")
    elif body.action in (ReviewStatus.approved, ReviewStatus.rejected, ReviewStatus.revision_requested):
        if record.review_status != ReviewStatus.pending_review:
            raise HTTPException(400, f"只有待复核状态才能执行 {body.action}")
    else:
        raise HTTPException(400, f"不支持的 action: {body.action}")
    ...
```

#### 4.3 权限矩阵

```python
# api/permissions.py

class Action(str, Enum):
    case_create        = "case_create"
    case_view          = "case_view"
    case_list          = "case_list"
    material_add       = "material_add"
    extraction_trigger = "extraction_trigger"
    analysis_trigger   = "analysis_trigger"
    review_submit      = "review_submit"   # 提交待复核（律师）
    review_decide      = "review_decide"   # 批准/打回（复核员）
    export_case        = "export_case"
    admin_users        = "admin_users"

# 权限矩阵：role → set[Action]
PERMISSIONS: dict[UserRole, set[Action]] = {
    UserRole.admin: set(Action),                              # 全部
    UserRole.senior_lawyer: {
        Action.case_create, Action.case_view, Action.case_list,
        Action.material_add, Action.extraction_trigger,
        Action.analysis_trigger, Action.review_submit, Action.export_case,
    },
    UserRole.junior_lawyer: {
        Action.case_create, Action.case_view, Action.case_list,
        Action.material_add, Action.extraction_trigger,
    },
    UserRole.reviewer: {
        Action.case_view, Action.case_list,
        Action.review_decide, Action.export_case,
    },
    UserRole.readonly: {
        Action.case_view, Action.case_list,
    },
}

def require_permission(action: Action) -> Callable[[UserContext], UserContext]:
    """FastAPI 依赖工厂：注入 UserContext，检查角色权限。
    返回值是一个 FastAPI 兼容的依赖函数，其参数为 UserContext（由 get_current_user 提供）。

    用法：
        @app.post("/api/cases/")
        async def create_case(
            body: CreateCaseRequest,
            user: UserContext = Depends(require_permission(Action.case_create)),
        ): ...
    """
    def _check(user: UserContext = Depends(get_current_user)) -> UserContext:
        if action not in PERMISSIONS.get(user.role, set()):
            raise HTTPException(status_code=403, detail=f"权限不足: {action}")
        return user
    return _check
```

```python
    reviews: list[Any] = Field(default_factory=list)

class BulkExportRequest(BaseModel):
    # Pydantic v2 注意：Field(max_length=N) 作用在 list 上约束的是每个字符串元素的长度，
    # 不是列表元素数量。限制列表长度必须用 conlist(str, max_length=50)。
    case_ids: conlist(str, max_length=50)    # from pydantic import conlist
    format: ExportFormat = ExportFormat.json
    include_materials: bool = False          # 原始材料文本，默认不含（审计合规）
```

> **`conlist` vs `Field(max_length=50)`（Pydantic v2 关键差异）**：`Field(max_length=50)` 对 `list[str]` 生效时约束的是每个字符串元素的字符数，不是列表元素数量。限制批量导出最多 50 个案件必须使用 `conlist(str, max_length=50)`（`from pydantic import conlist`）。务必在 `test_export_endpoints.py` 中写一个提交 51 个 `case_ids` 返回 422 的用例验证约束生效。

#### 5.3 端点设计

```
# 单案件结构化 JSON 导出
GET /api/cases/{case_id}/export?format=json
→ 200, application/json, Content-Disposition: attachment; filename=case_{id}.json

# 单案件 DOCX 导出（已有 report 端点的重命名别名）
GET /api/cases/{case_id}/export?format=docx
→ 302 / 200, 重定向或直接返回 report.docx

# 单案件 Markdown 导出
GET /api/cases/{case_id}/export?format=markdown
→ 200, text/markdown

# 批量导出（ZIP）
POST /api/cases/export/bulk      ← 注意：路径中 "export" 是字面量段
Body: {
  "case_ids": ["case-aaa", "case-bbb"],
  "format": "json"   // "json" | "docx" | "markdown"
}
→ 200, application/zip, report_export_{timestamp}.zip
   ZIP 结构：
     export_{timestamp}/
       case_aaa/case_aaa.json
       case_bbb/case_bbb.json
```

#### ZIP 包内部布局

每个案件在 ZIP 中独立子目录，布局如下：

```
case_{id}/
  case_meta.json          # CaseSnapshot JSON（顶层产物）
  artifacts/
    evidence_index.json
    issue_tree.json
    analysis_data.json
    report.md
    report.docx           # 若存在
  events.jsonl            # 审计事件日志
```

> **路由注册顺序（关键）**：FastAPI 按注册顺序匹配路由。`POST /api/cases/export/bulk` 中的 `export` 是字面量路径段，而 `POST /api/cases/{case_id}/...` 中 `{case_id}` 是参数段。若参数路由先注册，FastAPI 会将 `export` 当成 `case_id` 匹配，导致错误路由。
>
> **必须在 `app.py` 中将 `/api/cases/export/bulk` 路由注册在所有 `/api/cases/{case_id}/...` 路由之前**。
>
> 如果因代码组织原因无法保证顺序（例如路由在不同模块中注册），换用不含 `{case_id}` 歧义的路径：`POST /api/export/bulk`。

## 依赖与风险

### Phase 依赖图

```
Phase 1 (案件列表)
    │
    ├─► Phase 2 (审计日志)   ← 独立，可并行开发
    │
    ├─► Phase 3 (人工复核)   ← 依赖 Phase 1 的 review_status 集成到列表
    │       │
    │       └─► Phase 4 (用户权限)  ← review_decide 权限需要用户角色
    │               │
    │               └─► Phase 5 (导出增强) ← export 审计绑定 actor_id
    │
    └─► Phase 5 (导出增强)   ← export_json 组装不依赖 Phase 3/4
```

**关键约束**：
- Phase 3 的 `review_status` 字段需要写入 Phase 1 的 `CaseIndexEntry` 结构，故 Phase 3 必须在 Phase 1 之后
- Phase 5 的 `export_json` 可在 Phase 2 之后开发（需要 events）；但即使 Phase 2 未完成也可以先跳过 events 字段
- Phase 4 是 Phase 3 的自然后继（权限绑定 reviewer 角色），但 Phase 3 可以不等 Phase 4 独立上线（复核员暂时为 `"anonymous"`）

### 主要风险

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| `CaseStore` 线程安全：并发写入 `CaseIndex` | Phase 1 列表不一致 | `CaseIndex` 内部 `Lock`，测试覆盖并发 upsert |
| `events.jsonl` 并发追加行截断 | Phase 2 事件丢失 | `EventLog.append()` 内部持 `threading.Lock`（进程内线程安全）；单进程部署下此策略足够。加多线程并发写测试（Step 4b/4c）验证。 |
| JWT 密钥轮换：`API_SECRET_KEY` 变更导致所有 token 失效 | Phase 4 运行时全员登出 | 文档说明密钥轮换流程；可选：引入 `JWT_SECRET_KEY` 独立 env var |
| `users.json` 无加密存储 hashed_pwd | 安全风险 | Phase 4 用 bcrypt；生产部署说明文件权限（`chmod 600 users.json`） |
| ZIP 内存爆炸（批量导出超大案件） | Phase 5 OOM | 限制 `case_ids` 最多 50 个；单案件 JSON 导出无此问题 |
| `workspace_manager.py` 已有 640+ LOC | 修改风险高 | Phase 2 仅追加新方法，不修改已有方法；每次改动前重读文件（Rule 9） |

### 迁移注意事项

- **Phase 1**：存量案件的 `case_meta.json` 缺少 `created_at` 字段。`CaseIndex.scan_from_disk()` 需容错：缺失时使用 `workspace.json` 的文件 mtime 作为 fallback。
- **Phase 4**：未配置 `users.json` 时，`API_SECRET_KEY` 静态 Bearer 仍作为 fallback 保持向后兼容，避免已有 CI/集成脚本中断。
