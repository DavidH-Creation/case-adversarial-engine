# North Star

## 这是什么系统

多角色案件对抗推演系统是一个面向中国诉讼场景的案件备战系统。它的核心不是“替律师判案”，而是把案件材料结构化为可推演对象，并在严格的程序约束、证据规则和权限边界下，提前暴露争点、证据缺口、程序风险与可能的裁判路径。

这个仓库是该系统的长期 source of truth。后续代码、任务单、评测脚本和接口命名都必须从这里定义的产品边界和对象模型出发，而不是反向由实现细节决定系统形态。

## 这不是什么系统

- 不是 AI 法官，不输出“法院一定会怎么判”的对外承诺。
- 不是自由聊天 demo，不允许把多角色互喷当成法律推演。
- 不是终端用户法律咨询产品，首要用户不是普通个人。
- 不是自动对法院提交材料的流程机器人。
- 不是绕过律师复核直接生成正式法律意见的自动化系统。

## 目标用户

- 诉讼律师
- 庭前备战团队
- 律所内部复核人员
- 负责案件材料结构化、证据梳理和攻防推演的法律产品团队

## 当前优先级

已完成 `v1.5`（程序化庭前会议 / 质证版）：在 v1.2 基础上引入 `evidence_state_machine`、法官发问机制，已合并到 main。

默认案型固定为民事 `民间借贷`。优先原因不是市场口号，而是它在要件、证据、争点和举证责任上更适合做第一版结构化建模与评测闭环。

v1.2 在 v0.5（静态分析）和 v1（双边对抗）基础上，新增 12 个分析模块（P0.1–P0.4、P1.5–P1.8、P2.9–P2.12），覆盖：

- 争点影响排序与决策权重
- 金额一致性硬校验
- 裁判路径树与置信度区间
- 最强攻击链（原被告各一份）
- 证据权重评分与证据缺口 ROI 排序
- 争点分类与行动建议
- 可信度评分、替代主张生成、执行摘要

**当前目标**：`v2` — 民事通用内核 Beta（多案型支持：`civil_loan` / `labor_dispute` / `real_estate`）。

## 产品工作流原则

虽然 `v0.5` 还不做 `ui`，但系统未来统一按以下五阶段工作流组织：

1. `case_structuring`：案件结构化
2. `procedure_setup`：程序配置
3. `simulation_run`：对抗推演
4. `report_generation`：报告生成
5. `interactive_followup`：深度追问

这五阶段是产品体验层的统一骨架，不替代法律程序状态机。也就是说，前端工作流和后端法律回合都要存在，但不能混成一个状态体系。

## 借鉴与不借鉴

这套系统会借鉴异类多 agent 产品里的高价值机制，但只借鉴适合律师工作流的部分。

明确借鉴：

- 流程编排
- 状态管理
- 项目/案件上下文持久化
- 报告层与报告后互动层
- 可视化与运行回放

明确不借鉴：

- 社交平台动作空间
- 高自由人格生成
- 用“平行世界仿真”替代法律程序约束
- 对外部图谱/记忆平台的强耦合依赖

## 当前明确不做什么

- 不做法官裁决
- 不做多轮对抗
- 不做刑事与行政案种
- 不做 UI 和在线协作
- 不做面向 C 端的自助法律咨询
- 不做未经理师复核的正式意见导出

## 成功信号

- 系统能够稳定输出结构化争点、证据和举证责任映射。
- 结论能够回溯到明确的证据编号和来源。
- 文档、后续任务和代码实现对同一对象和状态使用完全一致的命名。
- 案件结果能够进入统一案件工作流，而不是每个阶段各自保存一套孤立产物。
