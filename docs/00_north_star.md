# North Star

## 系统定位

`case-adversarial-engine` 是一个给律师和案件备战团队使用的结构化案件推演系统。

它的工作不是替律师拍板，而是把材料、争点、证据、程序状态和推演结果放进同一套对象模型里，让人可以更早看到：

- 哪些争点真正决定结果
- 哪些证据链不完整
- 哪些程序节点会成为风险
- 对方最强的攻击路径是什么
- 当前最值得补强和解释的点是什么

## 目标用户

- 诉讼律师
- 庭前备战团队
- 律所内部复核人员
- 负责法律 AI 工具链的产品和工程团队

## 当前产品边界

当前主线是中国民事诉讼备战，不是泛法律聊天。

版本上分两条线：

- `Core Track = v2`
  关注统一对象模型、五阶段工作流、多案型 civil kernel、workspace persistence、scenario。
- `Output Track = v3.2`
  关注报告架构、Markdown / DOCX parity、用户可见语义和导出稳定性。

这两条线不是一回事，也不应该再混成一条单线版本历史。

## 核心原则

### 1. 先对象模型，后 prompt

系统首先是对象驱动的，不是聊天驱动的。后续代码、接口、任务单和测试必须服从 `docs/03_case_object_model.md`。

### 2. 先证据追溯，后结论表达

任何关键结论都必须能回到争点和证据，不接受“看起来很像结论”的黑箱段落。

### 3. 先程序边界，后角色发挥

角色化输出必须被程序状态、访问域和证据状态机约束，不允许自由发挥替代流程。

### 4. 先持久化和恢复，后新功能

能安全重跑、重连、恢复和比较，优先级高于新增一层“聪明输出”。

### 5. 先律师工作流，后 demo 观感

这套系统服务的是真实办案链路，不是社交化 agent 演示。

## 明确不做

- 不做 AI 法官
- 不做“胜诉概率”承诺型产品
- 不做面向公众的自助法律咨询
- 不做自动向法院提交材料
- 不做绕过律师复核的正式意见导出
- 不做社交平台式 action space

## 成功信号

- 案件可以稳定落到统一 `CaseWorkspace` / `Run` / `Job` / `ReportArtifact` 体系里。
- 同一案件从 CLI、API、scenario 和 report export 看出去，语义一致。
- 报告与追问不是孤立文本，而是可回放、可追溯的结构化产物。
- 新案型扩展时，系统扩的是插件和模板，不是推翻对象模型。
