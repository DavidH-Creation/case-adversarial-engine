# P0.1 争点影响排序模块设计

## 定位

case-adversarial-engine v1.2 P0 必须实现项之一。所属工作流阶段：`simulation_run`。

让每个争点从平铺列表升级为带决策权重的优先排序结构，告诉律师每个争点的决策价值、风险权重和优先行动。

依赖 P0.2 `AmountCalculationReport` 的输出，必须在 `amount_calculator` 运行完成后执行。

## 数据模型变更

### 新增枚举（`engines/shared/models.py`）

```python
class OutcomeImpact(str, Enum):
    high = "high"
    medium = "medium"
    low = "low"

class ImpactTarget(str, Enum):
    principal = "principal"
    interest = "interest"
    penalty = "penalty"
    attorney_fee = "attorney_fee"
    credibility = "credibility"

class EvidenceStrength(str, Enum):
    strong = "strong"
    medium = "medium"
    weak = "weak"

class AttackStrength(str, Enum):
    strong = "strong"
    medium = "medium"
    weak = "weak"

class RecommendedAction(str, Enum):
    supplement_evidence = "supplement_evidence"
    amend_claim = "amend_claim"
    abandon = "abandon"
    explain_in_trial = "explain_in_trial"
```

### `Issue` 模型扩展（向后兼容，全部 Optional）

5 个 spec 规定字段 + 1 个依据说明字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| `outcome_impact` | `Optional[OutcomeImpact]` | 对最终裁判结果的影响程度 |
| `impact_targets` | `list[ImpactTarget]` | 影响的诉请对象列表 |
| `proponent_evidence_strength` | `Optional[EvidenceStrength]` | 主张方当前证据强度（Burden.burden_party_id 确定角色） |
| `opponent_attack_strength` | `Optional[AttackStrength]` | 反对方在该争点的攻击强度 |
| `recommended_action` | `Optional[RecommendedAction]` | 系统建议行动 |
| `recommended_action_basis` | `Optional[str]` | 建议行动依据说明（spec 约束"必须附带依据说明"的持久化字段） |

## 数据流

```
输入
  IssueTree                  ← case_structuring 产物
  EvidenceIndex              ← case_structuring 产物
  AmountCalculationReport    ← P0.2 产物（必须先完成）
  proponent_party_id         ← 确定主张方角色（对应 Burden.burden_party_id）

      ↓ IssueImpactRanker.rank()

[LLM 批量评估 — 一次调用]
  system prompt: 角色约束 + 枚举约束 + 输出格式（JSON array）
  user prompt:   争点树摘要 + 证据索引摘要 + 举证责任方说明 + AmountConsistencyCheck 摘要

      ↓ LLMIssueEvaluationOutput（JSON array）

[规则层]
  1. 解析枚举：非法值 → 清空该字段，记入 unevaluated_issue_ids
  2. 过滤非法 issue_id / evidence_id（仅保留争点树和证据索引中已知的 ID）
  3. 校验证据绑定：proponent/opponent strength 非 None 时 issue.evidence_ids 必须非空
  4. 校验 recommended_action_basis：recommended_action 非 None 时 basis 必须非空
  5. 降级：校验失败的争点清空对应评估字段，记入 unevaluated_issue_ids
  6. 排序：outcome_impact DESC → opponent_attack_strength DESC → 原始顺序（稳定）

输出
  IssueImpactRankingResult
    ranked_issue_tree: IssueTree   ← issues 已富化 + 已排序
    evaluation_metadata: dict      ← model/timestamp/token 元信息
    unevaluated_issue_ids: list    ← 未能评估的争点 ID（供审计）
```

## 排序规则（纯规则层，无 LLM）

```python
IMPACT_ORDER  = {OutcomeImpact.high: 0, OutcomeImpact.medium: 1, OutcomeImpact.low: 2}
ATTACK_ORDER  = {AttackStrength.strong: 0, AttackStrength.medium: 1, AttackStrength.weak: 2}

key = (
    IMPACT_ORDER.get(issue.outcome_impact, 99),         # None 排末尾
    ATTACK_ORDER.get(issue.opponent_attack_strength, 99),
    original_index,                                      # 保证稳定性
)
```

主排序：`outcome_impact` 降序（high → medium → low）
次排序：`opponent_attack_strength` 降序（strong → medium → weak）
未评估争点排最后，内部维持原始顺序。

## AmountConsistencyCheck 集成

在 user prompt 中注入以下文本块（规则层不强制覆盖 LLM 判断，保留 LLM 决策空间）：

```
【金额一致性状态】
- verdict_block_active: {True/False}
- principal_base_unique: {True/False}
- all_repayments_attributed: {True/False}
- unresolved_conflicts 数量: {N} 条

注意：若 verdict_block_active=True，与本金、利息、违约金相关的争点
通常应评估为 outcome_impact=high。
```

## 合约保证

| 约束 | 强制层 |
|------|--------|
| `outcome_impact` / `recommended_action` 必须使用枚举值 | 规则层（非法值清空） |
| `proponent/opponent_strength` 非 None 时 `evidence_ids` 非空 | 规则层（失败则清空） |
| `recommended_action` 非 None 时 `recommended_action_basis` 非空 | 规则层（失败则清空） |
| 输出 `issues` 按 `outcome_impact` DESC 排序 | 规则层 |
| LLM 返回未知 `issue_id` 忽略 | 规则层（过滤） |
| LLM 调用失败返回 failed 结果，不抛异常 | ranker（try/except） |

## 测试策略

- **单元测试**（`test_ranker.py`）：不依赖 LLM，使用 stub LLM client
  - 排序规则：多种 outcome_impact / opponent_attack_strength 组合
  - 枚举降级：非法枚举值 → 清空字段，记入 unevaluated_issue_ids
  - 证据绑定校验：strength 非 None 但 evidence_ids 为空 → 清空
  - recommended_action_basis 校验
  - AmountConsistencyCheck 注入：验证 prompt 包含正确的 verdict_block_active 值
  - LLM 调用失败：返回 failed 结果，issues 保持原顺序，所有评估字段为 None
  - 空争点树：正常返回空 ranked_issue_tree

- **集成测试**：不在本模块范围内（由 simulation_run 顶层集成测试覆盖）
